● This is a starter/scaffold project for a Flask expense tracker web app — it
  looks like a teaching project where students fill in the implementation step
  by step.

  Here's what's in place so far:

  - Flask web app (app.py) with routes already defined for landing, register,
  login, logout, profile, and CRUD operations for expenses
  - SQLite database layer (database/db.py) — currently just a comment stub;
  students are meant to implement get_db(), init_db(), and seed_db()
  - HTML templates for landing, register, and login pages
  - Static CSS/JS files

  The routes are broken into numbered steps (Step 1–9), most returning
  placeholder strings. The full feature set when complete would be: user
  registration/login/logout, a profile page, and adding/editing/deleting
  expenses — a standard personal finance tracker.

● Here's the full project structure explained:

  expense-tracker/
  ├── app.py                  # Entry point — Flask app + all route definitions
  ├── requirements.txt        # Python dependencies
  ├── .gitignore
  │
  ├── database/
  │   ├── __init__.py
  │   └── db.py               # Stub for SQLite helpers (get_db, init_db,
  seed_db) — not yet implemented
  │
  ├── templates/              # Jinja2 HTML templates
  │   ├── base.html           # Master layout — navbar, footer, font/CSS/JS
  links
  │   ├── landing.html        # Public homepage (extends base.html)
  │   ├── login.html          # Sign-in page (extends base.html)
  │   └── register.html       # Sign-up page (extends base.html)
  │
  └── static/
      ├── css/
      │   └── style.css       # All custom styles
      └── js/
          └── main.js         # Empty — students add JS as features are built

  How it fits together:

  - base.html is the shared shell — every other template extends it using
  Jinja2's {% extends %} / {% block %} system. It provides the navbar and
  footer.
  - app.py maps URLs to Python functions, which render the templates.
  - database/db.py is meant to hold the SQLite connection logic — it's the only
  backend layer, no ORM involved.
  - static/ holds files served directly to the browser without Flask processing
  them.

  The structure follows a simple MVC-lite pattern: app.py is the controller,
  templates/ is the view, and database/db.py will be the model once implemented.
